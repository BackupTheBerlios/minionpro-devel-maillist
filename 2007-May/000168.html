Am 17.05.2007 um 13:28 schrieb Bernhard Enders:
> Another haze point, at least to me, is where to use the files:
> replace-*.sed and substitutions.list? They seem to be used nowhere in
> the compilation process. Maybe they are needed in the generation of
> encoding files? Thanks a lot,

These are files that I used for generating glyph-list*, they are  
currently not needed when generating your own set of metrics. In case  
future versions of MinionPro have different glyphnames these files  
remain in the CVS for convenience.

substitutions.list contains a table of glyphnames that differ between  
font versions, first row: v2000, second row: v1001, third row v1000.

 From this table the replace*.sed files were generated through
   cat substitutions.list | awk '$1!=$3 {print $1" "$3}' | sed -e  
's#^#s/\\\/#' -e 's# #\\\([ ]\\\|$\\\)/\\\/#' -e 's#$# /g#' >  
replace.v1000.sed
   cat substitutions.list | awk '$1!=$2 {print $1" "$2}' | sed -e  
's#^#s/\\\/#' -e 's# #\\\([ ]\\\|$\\\)/\\\/#' -e 's#$# /g#' >  
replace.v1001.sed

The replace*sed files can then be used to convert v2000 enc files  
into that of another font version. The advantage of this approach is  
that now the same set of metrics can be used with all font versions,  
just the enc need to be replaced -- and it works with both packed raw  
encodings and autogenerated encodings. The latter isn't possible if  
otftotfm is run on different encodings, because for example  
automatically allocated encoding slots for f_t and T_h vary between  
font versions (due to different ordering of the lookups in the otf, I  
suppose).

The question that might remain: Where does substitution.list come  
from? This was some perl processing of Andreas, he compared the sets  
of autogenerated .enc files (one set for each font version).
