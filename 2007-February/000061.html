Hello,

> Thanks for the list. But in order to understand it correctly, what 
> exactly is printed in column 1,2,3? Is it v2000 - v1001 - v1000?

in this case, it's v2000 on the left of the => and v1000 on the right.
This is what I did: I used the MinionPro.map files (of v2000 and v1000
in this case) to find the pairs of enc files that should be mapped (used
for the same TeX font) and derived pairs of matching glyphs from these
(file "map-2000-1000" in my example, which looks like this):

/a      /a
/A      /A
/aacute /aacute
/Aacute /Aacute
/aacute.sc      /Aacutesmall
/Aacute.swash   /Aacute.swash
/abreve /abreve
/Abreve /Abreve
/abreve.sc      /uniF66D
...

(v2000 glyph, v1000 glyph)

In this list, you can find ambiguous mappings like

/zero.denominator       /uniF655
/zero.denominator       /uniF661
/zero.numerator /uniF655
/zero.numerator /uniF661

which lead to the warnings shown:

> > WARNING: /zero.denominator => /uniF655 /uniF661
> > WARNING: /zero.numerator => /uniF655 /uniF66

> > Of course, there could be many more (non-ambiguous) wrong mappings.)
> 
> Why?

Well, there could be cases where glyphs /A.2000 =~ /A.1000 and /B.2000
=~ /B.1000 are allocated in the order /A.2000 /B.2000 in one encoding
and in the order /B.1000 /A.1000 in the other (of course, this only
happens if none of the glyphs is fixed to a certain encoding slot). Then
my list would contain

/A.2000 /B.1000
/B.2000 /A.1000

and if these glyphs are not used in a different way in another encoding,
the mapping would be wrong but not ambiguous. Or isn't this possible?
